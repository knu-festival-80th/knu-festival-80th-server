## 1. 성능 (Performance)

### NFR-PERF-01: 응답 시간

| 항목 | 목표 |
|------|------|
| 일반 API (CRUD) | 평균 200ms 이하, p95 500ms 이하 |
| 랭킹/목록 조회 API | 평균 100ms 이하 (캐시 히트 시) |
| 이미지 업로드 | 3초 이하 (5MB 기준) |
| WebSocket 메시지 전달 | 100ms 이하 (서버 처리 시간 기준, 네트워크 레이턴시 제외) |

### NFR-PERF-02: 동시 접속

| 항목 | 목표 |
|------|------|
| 동시 접속 사용자 | 최소 500~800명 |
| WebSocket 동시 연결 (캔버스) | 최소 200 연결 |
| 초당 요청 처리량 (RPS) | 최소 300~500 RPS |

> **근거**: 축제 피크 시간대(공연 전후) 실제 동시 접속률은 전체 방문객의 3~5% 수준. 단일 인스턴스 기준이며, 수평 확장 시 비례 증가.

### NFR-PERF-03: 데이터 캐싱

- 부스 랭킹 데이터: Redis Sorted Set, 실시간 갱신
- 공연 정보: Redis 캐시, TTL 5분, 변경 시 무효화
- 피드/부스 좋아요 카운트: Redis 카운터, 주기적 DB 동기화
- 대기열 현황: Redis 캐시, TTL 10초

## 2. 가용성 (Availability)

### NFR-AVAIL-01: 가동 시간

| 항목 | 목표 |
|------|------|
| 축제 기간 중 서비스 가용성 | 99.5% 이상 |
| 피크 구간 (18시~자정) | 별도 모니터링 강화, 무장애 운영 목표 |
| 계획된 점검 | 축제 기간 중 금지, 사전에 완료 |

### NFR-AVAIL-02: 장애 대응 (Resilience)

| 장애 상황 | 대응 정책 |
|-----------|-----------|
| 서버 프로세스 종료 | 자동 재시작 (프로세스 매니저) |
| DB 커넥션 풀 고갈 | 핵심 기능(대기열) 우선, 비핵심 기능 일시 차단 |
| SMS API 장애 | 대기열 관리 정상 유지, SMS만 실패 처리 + 관리자 재발송 |
| Redis 장애 | DB 직접 조회로 Fallback (성능 저하 허용), 좋아요 카운트는 최대 동기화 주기만큼 오차 허용 |
| WebSocket 연결 끊김 | 클라이언트 자동 재연결 + REST API로 현재 상태 복구 |

## 3. 보안 (Security)

### NFR-SEC-01: 인증/인가

- 관리자 API (`/admin/**`): JWT 토큰 기반 인증 필수
- 사용자 API (`/api/**`): 인증 불필요 (공개 API)
- JWT 만료: Access Token 1시간, Refresh Token 7일
- 관리자 역할 분리: SUPER_ADMIN / BOOTH_ADMIN (부스별 소유권 검증)

### NFR-SEC-02: 데이터 보호

- 전화번호: DB 저장 시 암호화 처리
- 매칭 비밀번호: 단방향 해시 저장
- 관리자 비밀번호: BCrypt 해시 저장
- 환경변수(.env): Git에 포함하지 않음
- HTTPS 필수 (프로덕션 환경)
- CORS 허용 도메인 화이트리스트 적용

### NFR-SEC-03: 입력 검증 및 남용 방지

- 모든 API 입력값: 서버 사이드 검증 (Bean Validation)
- 파일 업로드: 허용 확장자 (jpg, png, webp), 최대 10MB, Content-Type 검증
- Rate Limiting 적용 대상:
  - 관리자 로그인: 실패 시 IP 기반 제한
  - 리뷰 작성: IP 기반 횟수 제한
  - 피드 등록: IP 기반 횟수 제한
  - 매칭 결과 조회: 실패 시 IP 기반 제한 (brute force 방지)
  - 좋아요 API: 세션 토큰 기반 중복 방지

## 4. 확장성 (Scalability)

### NFR-SCALE-01: 수평 확장

- Stateless 서버 설계 (세션은 JWT, 상태는 Redis/DB)
- WebSocket 멀티 인스턴스 시 Redis Pub/Sub 메시지 브로드캐스트
- 파일 업로드는 외부 스토리지(S3)로 분리

## 5. 유지보수성 (Maintainability)

### NFR-MAINT-01: 코드 품질

- CQRS-lite 패턴 준수 (Command/Query 서비스 분리)
- 레이어드 아키텍처 (Presentation → Application → Domain → Infra)
- CLAUDE.md에 정의된 코드 컨벤션 준수

### NFR-MAINT-02: 로깅 및 모니터링

- 요청/응답 로그: 구조화된 포맷
- 에러 로그: 스택 트레이스 포함
- 민감 정보(전화번호, 비밀번호): 로그에서 마스킹
- 모니터링: Spring Actuator health/metrics 엔드포인트 노출
- 장애 알림: Slack Webhook 또는 동등한 수단으로 즉시 알림

### NFR-MAINT-03: API 문서화

- Swagger/OpenAPI 3.0 기반 자동 문서 생성
- 각 컨트롤러별 `*Docs` 인터페이스에 Swagger 어노테이션 분리 작성

## 6. 호환성 (Compatibility)

### NFR-COMPAT-01: 클라이언트

- 모든 API는 RESTful JSON 응답
- WebSocket: STOMP 프로토콜 (캔버스 실시간 통신)
- 모바일 브라우저 (iOS Safari, Android Chrome) 최적화 대상

### NFR-COMPAT-02: 외부 서비스 의존성

| 외부 서비스 | 용도 | 장애 시 영향 | 대응 |
|-------------|------|-------------|------|
| SMS API | 대기 호출 알림 | 알림 불가 | 대기열 관리 정상, 관리자 재발송 |
| AWS S3 | 이미지 저장 | 업로드 불가 | 기존 이미지는 CDN에서 제공 |
| Redis | 캐시, 실시간 데이터 | 성능 저하 | DB 직접 조회 Fallback |

## 7. 데이터 관리

### NFR-DATA-01: 소프트 딜리트

- 사용자 데이터가 포함된 테이블: 소프트 딜리트 (`deleted_at` 타임스탬프)
- 대기열(waiting) 포함: 전화번호 등 개인정보 보호

### NFR-DATA-02: 백업

- DB 일일 백업 (축제 기간 중 6시간 주기)
- 백업 데이터 보관: 축제 종료 후 90일

### NFR-DATA-03: 축제 종료 후 처리

| 시점 | 처리 내용 |
|------|-----------|
| D+1일 | 서비스 종료 안내 공지 게시 |
| D+7일 | 전화번호 마스킹 배치 실행 |
| D+30일 | 소프트 딜리트된 개인 데이터 물리 삭제 |
| D+90일 | S3 이미지 정리, DB 백업 보관 만료 |
