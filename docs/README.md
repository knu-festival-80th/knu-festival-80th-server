# 문서 체계 가이드

> 2026 경북대학교 80주년 대동제 웹앱 서비스 (백엔드)

## 문서 구조

```
docs/
├── 01-requirements-spec.md   ← 사용자 요구사항 명세서 (SRS)
├── 02-functional-spec.md     ← 기능 명세서 (FS)
├── 03-non-functional-spec.md ← 비기능적 요구사항 명세서 (NFR)
└── 04-technical-spec.md      ← 기술/개발 명세서 (TS)
```

## Validation vs Verification

| | Validation (확인) | Verification (검증) |
|---|---|---|
| **질문** | 올바른 제품을 만들고 있는가? | 제품을 올바르게 만들고 있는가? |
| **관점** | 사용자/이해관계자 | 개발팀/기술 |
| **기준 문서** | 01-requirements-spec.md | 04-technical-spec.md |
| **검증 방법** | 인수 테스트, 사용자 시연 | 단위 테스트, 통합 테스트, 코드 리뷰 |

## 각 문서의 역할

| 문서 | 독자 | 핵심 질문 | 내용 |
|------|------|-----------|------|
| **01 SRS** | 기획자, 전체 팀 | 사용자가 원하는 게 이게 맞나? | 사용자 스토리, 인수 조건 |
| **02 FS** | 백엔드, 프론트엔드 | 어떤 API를 만들어야 하나? | API 엔드포인트, 비즈니스 규칙 |
| **03 NFR** | 개발팀, DevOps | 어떤 품질 수준을 유지해야 하나? | 성능, 보안, 가용성, 장애 대응 |
| **04 TS** | 백엔드 개발팀 | 어떤 기술로 어떻게 만드나? | 기술 스택, DB 스키마, 설정 |

## 요구사항 추적 흐름

```
사용자 니즈 → [01 SRS] → [02 FS] → [04 TS] → 코드 구현
                ↑                        ↑
            Validation              Verification
           (인수 테스트)           (단위/통합 테스트)
                
                         [03 NFR]
                            ↑
                     품질 속성 검증
                    (성능/보안 테스트)
```

## 문서 갱신 규칙

### 언제 문서를 갱신해야 하는가
- 사용자 요구·인수 조건 변경 → `01-requirements-spec.md`
- API 엔드포인트·요청/응답·비즈니스 규칙 변경 → `02-functional-spec.md`
- 성능·가용성·보안·데이터 보호 등 품질 속성 변경 → `03-non-functional-spec.md`
- 기술 스택·DB 스키마·인덱스·캐싱·인증 구조 변경 → `04-technical-spec.md`

다음의 경우는 문서 갱신이 필요 없다:
- 외부 계약이 바뀌지 않는 내부 리팩토링
- 명세에 반영되지 않은 내부 구현 디테일의 수정
- 문서에 영향 없는 단순 버그 수정

### 변경 이력 테이블 작성 규칙

각 문서 상단의 **변경 이력** 테이블에 새 행을 추가한다.

- 컬럼 고정: `버전 | 날짜 | 변경 내용 | 변경자`
- 날짜는 `YYYY-MM-DD` 형식
- 변경자는 GitHub 핸들 또는 이름. 모르면 `-`
- 변경 내용은 한 줄, 무엇이 어떻게 바뀌었는지 핵심만

### 버전 증가 기준 (Semantic Doc Versioning)

| 변경 유형 | 버전 증가 | 예시 |
|---|---|---|
| 표현·오타·서식만 | patch (`v1.2` → `v1.2.1`) | 문구 다듬기, 표 정렬 |
| API·규칙·스키마 추가/변경/삭제 | minor (`v1.2` → `v1.3`) | 엔드포인트 추가, 비즈니스 규칙 신설 |
| 문서 구조 개편 | major (`v1.2` → `v2.0`) | 섹션 재편, 도메인 분리·통합 |

### 운영 흐름
- 문서와 코드는 같은 PR에서 함께 갱신한다
- 코드 PR을 만들 때 변경된 도메인의 명세 갱신 여부를 셀프 체크한다
- 명세에 반영되지 않은 임시 결정은 기록하지 말고, 결정이 굳어지면 그때 문서로 옮긴다
